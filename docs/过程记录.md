# 过程记录

## 问题定位

关卡策划（LD）用乐高式白盒搭好场景后，场景美术（LA）需要把每个方体变成对应的 placeholder 模型（PH）。这一步不需要精致建模，是纯体力活。一个中等场景可能有上百个白盒物件，传统手工 PH 制作消耗大量人时。

## 决策记录

### 决策1：Godot Editor Plugin vs 独立工具
选择 **Editor Plugin**。因为 LD/LA 的工作流已经在 Godot 里——白盒搭在场景中，PH 也要放回场景中。独立工具会打断工作流，增加导入导出摩擦。
- 代价：限定 Godot 4.6+，不能给 Unity/Unreal 团队直接用。
- 但 Godot 社区足够大，且项目使用 Godot，这个限定是合理的。

### 决策2：LLM 只解析不生成 Mesh
让 LLM 把描述翻译成 `[几何体类型 + 尺寸 + 位置 + 颜色]` 的 JSON，具体 Geometry 由代码程序化生成。理由是：
- LLM 不擅长生成精确的 3D 顶点数据
- JSON 可审查、可手工修改，用户保持控制权
- 生成结果确定可复现
- 离线模式下可以 fallback 到预设模板库

### 决策3：Programmatic Native UI vs .tscn
选用纯代码构建 UI 面板（`_build_ui()` 函数），不使用 .tscn 场景文件。减少依赖、加快开发迭代。

### 决策4：双格式输出（.glb + .fbx）
.glTF 是开放标准，Godot 原生支持；.fbx 通过 Python+trimesh 转换。优先保证 .glb 一定能出，.fbx 作为可选项。这样兼顾通用性和生态兼容性。

## AI 协作过程

1. **需求沟通**：与 AI 多轮对齐"白盒→PH"这个具体痛点，AI 初步建议做 "Asset Placeholder Toolkit"，经讨论调整为"AI 驱动的程序化 PH 生成器"。
2. **架构设计**：AI 建议用 EditorPlugin + Dock UI 方案，LLM 解析 + 程序化生成的二段式架构，我认可了核心思路。
3. **编码**：全部代码由 AI 生成，我负责 Review 逻辑正确性、调整 Prompt 中 Few-shot 示例的质量。
4. **调优**：Few-shot 示例增加到了 5 个（含轿车/木箱/掩体/木门/石柱），预设模板增加到 10 个物件类型。

## 学习收获

- Godot 4 `SurfaceTool` + `ArrayMesh` 程序化建模
- Godot `EditorPlugin` API 与 Dock 集成
- OpenAI 兼容 API 的 `response_format: json_object` 模式
- System Prompt + Few-shot 设计实践
